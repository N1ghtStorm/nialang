# Квантовые инструкции и конструкции NiaLang

Документ описывает, **какие квантовые примитивы нужны языку NiaLang**, чтобы блок
`quant { ... }` перестал быть пустой областью видимости и начал лоуэриться в
настоящий QIR (Quantum Intermediate Representation).

Документ — проектный (design spec), а не отчёт о реализованном. Он фиксирует
целевой набор типов, инструкций, операторов и правил семантики, на которые должны
ориентироваться `src/backend/qir`, `src/semantics/typecheck` и парсер.

## 1. Где мы сейчас

- `quant { ... }` и `gpu { ... }` распознаются лексером/парсером и в AST
  представлены как `Stmt::Quant` / `Expr::Quant` (см. `src/ast/mod.rs`). Сейчас они
  ведут себя как обычный блок-скоуп: внутренние `let` не утекают наружу, а тип
  выражения равен типу хвоста блока.
- Семантика дополнительно запрещает `return` и `break` внутри `quant`-выражения
  (`src/semantics/typecheck/mod.rs`).
- Бэкенд `Backend::Qir` (флаги `-q` / `--qir`) — **заглушка**
  (`src/backend/qir/mod.rs`): эмитит фиксированный QIR-модуль с opaque-типами
  `%Qubit` / `%Result`, объявлениями `__quantum__qis__h__body`,
  `__quantum__qis__x__body`, `__quantum__qis__cnot__body`,
  `__quantum__qis__mz__body`, `__quantum__rt__result_record_output`, пустым
  `@Main` и QIR module flags (`qir_major_version`, базовый профиль).
- Целевой формат лоуэринга — **QIR** (см. `QIR_REPOS_FOR_AI_CONTEXT/qir-spec`),
  профили **Base** и **Adaptive**.

> **Замечание про стиль заглушки.** Текущий стаб написан в старом стиле QIR:
> типизированные указатели `%Qubit*` / `%Result*` и `qir_major_version = 1`.
> Современная спецификация (примеры в `Base_Profile.md`) использует **opaque
> указатели `ptr`** и `qir_major_version = 2`. При реальном лоуэринге нужно
> перейти на `ptr`; типы `%Qubit` / `%Result` остаются лишь семантической
> абстракцией на уровне языка, а в IR кубит/результат — это `ptr`.

Задача документа — описать путь от «скоуп-заглушки» к реальному квантовому ядру.

## 1.1. Два уровня: язык Nia и QIR-профиль

Это ключевое различение, на котором держится весь документ. Их нельзя путать.

- **Уровень языка Nia.** Высокоуровневый, удобный для записи синтаксис: `mz(q)`
  «возвращает» `result`, в `quant` можно завести `let`, посчитать угол поворота,
  написать `if`. Это то, что видит пользователь и проверяет типчекер.
- **Уровень QIR-профиля.** Жёстко ограниченный целевой IR. **Base Profile**
  вообще не позволяет классические вычисления и локальные переменные: «It is not
  possible to express classical computations, such as adding two double values…
  defining local variables is not supported» (`Base_Profile.md`). Классическая
  арифметика, ветвление и т.п. — это возможности **Adaptive Profile** (причём
  арифметика — его *опциональная* возможность, см. §7).

Следствие: одна и та же программа на Nia может **компилироваться под Base**, если
её квантовая часть укладывается в ограничения (линейная схема, измерение в конце,
вывод через `record`), и **требовать Adaptive**, если использует ветвление по
измерению или классические вычисления внутри `quant`. Компилятор обязан
проверять, что выбранный профиль (`--qir-profile`) допускает использованные
конструкции, и отклонять несовместимые программы (см. §6 и §7).

## 2. Базовые типы

Язык должен получить два новых встроенных непрозрачных (opaque) типа, отражающих
QIR-сущности «кубит» и «результат измерения». В современном QIR оба
представлены как opaque-указатель `ptr` (типы `%Qubit` / `%Result` —
историческая запись); в Base Profile конкретный кубит/результат — это целая
константа, приведённая к `ptr` через `inttoptr (i64 N to ptr)`.

| Тип Nia | QIR (IR) | Семантика |
|---------|----------|-----------|
| `qubit` | `ptr` (идентификатор кубита) | Линейный ресурс — кубит. Нельзя копировать, нельзя возвращать наружу `quant`. |
| `result` | `ptr` (идентификатор результата) | Классический результат измерения (значение `Zero` / `One`). |

Требования к типам:

- `qubit` — **линейный (move-only) ресурс**. Его нельзя присвоить дважды,
  передать в две инструкции «по значению» одновременно или скопировать через
  `let b = a;`. Это нужно, чтобы статически отражать no-cloning theorem.
- `result` — обычное классическое значение; его можно сравнивать (`== One`,
  `== Zero`), печатать и использовать в `if`.
- Оба типа существуют **только внутри `quant`-контекста** (см. §6). Вне `quant`
  переменная типа `qubit` не может быть объявлена.

### Регистры кубитов

Нужен агрегат для нескольких кубитов фиксированного размера, согласованный с уже
имеющимся синтаксисом массивов/векторов:

```nia
quant {
    let q: qubit[4];        // регистр из 4 кубитов (статический размер)
    h(q[0]);
    cx(q[0], q[1]);
}
```

- Статический размер `qubit[N]` транслируется в `N` статически пронумерованных
  кубитов `inttoptr (i64 0 to ptr)` … `inttoptr (i64 N-1 to ptr)` (Base Profile
  требует статической нумерации; см. `Base_Profile.md`, раздел *Data Types and
  Values*).
- Индексация `q[i]` отдаёт ссылку на один кубит, не копируя ресурс.

## 3. Выделение и освобождение кубитов

Две модели, соответствующие QIR module flags `dynamic_qubit_management`:

1. **Статическая (Base Profile, по умолчанию).** Кубиты нумеруются на этапе
   компиляции; число `required_num_qubits` подсчитывается компилятором и
   проставляется в атрибут entry-point. Явного `alloc`/`free` нет — кубиты живут
   на протяжении всего `quant`-блока.
2. **Динамическая (Adaptive Profile, опционально).** Нужны конструкции:

```nia
let q = qubit();          // __quantum__rt__qubit_allocate -> ptr
release(q);               // __quantum__rt__qubit_release
let reg = qubit_array(n); // __quantum__rt__qubit_array_allocate (динамический n)
```

Точные сигнатуры (см. `Memory_Management.md`):

```llvm
declare ptr  @__quantum__rt__qubit_allocate(ptr %out_err)
declare void @__quantum__rt__qubit_release(ptr %qubit)
declare void @__quantum__rt__qubit_array_allocate(i64 %N, ptr %array, ptr %out_err)
```

> Имя именно `qubit_array_allocate` (а не `qubit_allocate_array`). Bulk-аллокация
> требует включённого флага `"arrays"` **и** `"dynamic_qubit_management"`, а сам
> буфер массива (классическая память) выделяет вызывающая сторона (`alloca`).

На первом этапе достаточно **только статической модели** — она проще и покрывает
Base Profile. Динамику стоит вводить отдельным флагом профиля (только Adaptive).
`release` существует **только** в динамической модели; в Base Profile его нет —
кубиты живут весь блок и не освобождаются явно.

## 4. Набор квантовых инструкций (QIS)

Минимально необходимый набор, отображаемый на `__quantum__qis__*__body`. Имена в
Nia — встроенные функции (builtins), доступные внутри `quant`.

### 4.1. Однокубитные гейты без параметров

| Nia | QIR intrinsic | Описание |
|-----|---------------|----------|
| `h(q)` | `__quantum__qis__h__body` | Адамар |
| `x(q)` | `__quantum__qis__x__body` | Pauli-X (NOT) |
| `y(q)` | `__quantum__qis__y__body` | Pauli-Y |
| `z(q)` | `__quantum__qis__z__body` | Pauli-Z |
| `s(q)` | `__quantum__qis__s__body` | Фаза S (√Z) |
| `s_adj(q)` | `__quantum__qis__s__adj` | S† |
| `t(q)` | `__quantum__qis__t__body` | Фаза T (√S) |
| `t_adj(q)` | `__quantum__qis__t__adj` | T† |

### 4.2. Однокубитные параметрические повороты

Угол — значение типа `f64` (радианы).

| Nia | QIR intrinsic | Описание |
|-----|---------------|----------|
| `rx(theta, q)` | `__quantum__qis__rx__body` | Поворот вокруг X |
| `ry(theta, q)` | `__quantum__qis__ry__body` | Поворот вокруг Y |
| `rz(theta, q)` | `__quantum__qis__rz__body` | Поворот вокруг Z |

> **Ограничение профиля.** В Base Profile аргументы вызова обязаны быть
> константами («Call arguments must be constant values», `Base_Profile.md`).
> Поэтому угол должен быть **константой времени компиляции**: `rx(0.5, q)` — ок;
> `rx(a + b, q)` с вычисляемым углом требует Adaptive Profile с опциональной
> возможностью классических вычислений (§7).

### 4.3. Двух- и многокубитные гейты

| Nia | QIR intrinsic | Описание |
|-----|---------------|----------|
| `cx(ctrl, tgt)` / `cnot(ctrl, tgt)` | `__quantum__qis__cnot__body` | Controlled-NOT |
| `cz(ctrl, tgt)` | `__quantum__qis__cz__body` | Controlled-Z |
| `swap(a, b)` | `__quantum__qis__swap__body` | Обмен |
| `ccx(c1, c2, tgt)` | `__quantum__qis__ccx__body` | Тоффоли (CCNOT) |

### 4.4. Измерение и сброс

| Nia | QIR intrinsic | Сигнатура в IR | Описание |
|-----|---------------|----------------|----------|
| `mz(q) -> result` | `__quantum__qis__mz__body` | `void(ptr %q, ptr writeonly %r)` | Измерение в Z-базисе |
| `reset(q)` | `__quantum__qis__reset__body` | `void(ptr %q)` | Сброс кубита в \|0⟩ |
| `read_result(r) -> bool` | `__quantum__qis__read_result__body` | `i1(ptr %r)` | Чтение результата (Adaptive) |

> **Важно про `mz`: на уровне IR это `void`-функция, а не «возврат `result`».**
> Реальная сигнатура — `__quantum__qis__mz__body(ptr %qubit, ptr writeonly %result)`
> с атрибутом `irreversible` (`mz.ll`, `Base_Profile.md`). Результат — это
> **отдельная заранее пронумерованная сущность** (`inttoptr (i64 k to ptr)`),
> которую передают как `writeonly`-аргумент, а не возвращаемое значение.
>
> Синтаксис языка `let r = mz(q);` — это **сахар**: компилятор сам выделяет
> очередной номер результата `k`, увеличивает `required_num_results` и эмитит
> `call void @__quantum__qis__mz__body(ptr %q, ptr inttoptr (i64 k to ptr))`.
> Дальше `r` ссылается на результат `k`. В Base Profile прочитать значение `r`
> внутри программы нельзя — только записать в вывод через `record` (§5);
> `read_result` (получение `i1`) доступен только в Adaptive.

> Принцип: NiaLang предоставляет минимальный «канонический» набор как builtins, а
> производные гейты (например, контролируемые повороты) при необходимости
> выражаются через библиотеку на самом Nia, а не зашиваются в бэкенд — это
> согласуется с рекомендацией QIS из `Instruction_Set.md`.

## 5. Запись результата (output recording)

Программа сообщает рантайму, какие значения составляют вывод, через вызовы
`__quantum__rt__*_record_output`. Это **единственный** способ вернуть результат
наружу в Base Profile — возвращаемое значение entry-point это лишь `i64`-код
выхода (см. §6.1, §6a).

| Nia | QIR runtime | Сигнатура | Назначение |
|-----|-------------|-----------|------------|
| `record(r)` | `__quantum__rt__result_record_output` | `void(ptr %result, ptr %label)` | Записать один результат |
| `record_tuple(n)` | `__quantum__rt__tuple_record_output` | `void(i64 %n, ptr %label)` | Маркер начала кортежа из `n` элементов |
| `record_array(n)` | `__quantum__rt__array_record_output` | `void(i64 %n, ptr %label)` | Маркер начала массива из `n` элементов |

Ключевые уточнения (по `Base_Profile.md`):

- **Метка обязательна на уровне IR.** Второй `ptr`-аргумент — указатель на
  глобальную строковую константу с именем. Поэтому отдельного `record` без метки
  не существует: фронтенд **всегда** генерирует метку (бэкенд может её
  игнорировать в зависимости от output schema). На уровне языка можно дать
  `record(r)` (метка генерируется автоматически, например `"r0"`) и
  `record(r, "name")` (явная метка) — но в IR метка есть всегда.
- **Группировка вывода.** Чтобы вернуть несколько значений как единое целое,
  перед `record` ставится `record_tuple(n)` / `record_array(n)`. Например, два
  результата из `bell()` оформляются как кортеж: сначала
  `tuple_record_output(2, ...)`, затем два `result_record_output`.
- Все метки в пределах одного entry-point должны быть **уникальны**.
- Компилятор собирает вызовы `record*` в финальный блок и проставляет
  `required_num_results` в атрибутах entry-point.

## 6. Семантика блока `quant`

`quant { ... }` становится **квантовым контекстом**. Правила делятся на общие
(для всех профилей) и профиле-зависимые.

### 6.1. Общие правила (любой профиль)

1. Только внутри `quant` разрешено объявлять `qubit` / `qubit[N]` и вызывать
   квантовые builtins (§4).
2. **Линейность.** `qubit` — линейный ресурс: его нельзя копировать
   (`let b = a;`) или использовать дважды «по значению» в одном выражении.
   Это статически отражает no-cloning theorem.
3. После необратимой операции (`mz`) кубит **больше использовать нельзя** —
   ни как аргумент гейта, ни повторно измерять. Это прямое отражение правила QIR
   «Qubits must not be used after they have been passed as arguments to a
   function that performs an irreversible action» (`Base_Profile.md`).
4. **`qubit` нельзя вернуть наружу `quant`** (и наружу функции). Наружу могут
   уходить только классические значения, а вывод программы оформляется через
   `record` (§5), а не через возврат значения.

### 6.2. Правила, зависящие от профиля

| Правило | Base Profile | Adaptive Profile |
|---------|--------------|------------------|
| Локальные `let` для классических значений внутри `quant` | нет | да (опц. возможность «классические вычисления») |
| Классическая арифметика над углами/числами | нет (только константы) | да (опц.) |
| `if` по результату измерения (`read_result`) | нет | да |
| Измерение в середине схемы, использование кубита после | нет (измерение только в конце) | да |
| `release` кубита (динамическая модель) | нет | опц. |
| Освобождение кубита до конца блока | не требуется (живёт весь блок) | через `release` |

Текущее ограничение реализации — **`return` и `break` внутри `quant`-выражения
запрещены** (`src/semantics/typecheck/mod.rs`) — сохраняем: ранний выход из
квантового региона ломает статический подсчёт ресурсов в Base Profile.

### 6.3. Проверки компилятора

Типчекер должен:

- разрешать квантовые типы/builtins только в `quant`-контексте;
- следить за линейностью и запретом использования кубита после `mz`;
- знать **целевой профиль** (`--qir-profile`) и **отклонять** конструкции,
  недопустимые в нём (например, `if read_result(...)` или вычисляемый угол под
  Base Profile) — по аналогии с `docs/vector-limitations.md` для векторов.

### 6.4. Пример: Bell pair

На уровне **языка** запись остаётся высокоуровневой:

```nia
// Совместимо с Base Profile: только гейты, измерение в конце, вывод через record.
fn bell() {
    quant {
        let a: qubit;
        let b: qubit;
        h(a);
        cx(a, b);
        let ra = mz(a);     // сахар: выделяется результат №0
        let rb = mz(b);     // сахар: выделяется результат №1
        record_tuple(2);    // сгруппировать два результата в кортеж
        record(ra, "a");
        record(rb, "b");
    }
}
```

Здесь функция **ничего не возвращает** (вывод — через `record`), измерения идут в
конце, классических вычислений нет — это укладывается в Base Profile. Как это
ложится в IR — см. §6a.

## 6a. Целевой шаблон Base-Profile entry-point

Лоуэринг `quant`-блока под Base Profile должен порождать строго структурированный
entry-point из четырёх базовых блоков (`Base_Profile.md`, *Entry Point
Definition*). Это удобный «скелет», на который опирается бэкенд:

```llvm
define i64 @main() #0 {
entry:
  ; 1) инициализация среды
  call void @__quantum__rt__initialize(ptr null)
  br label %body

body:
  ; 2) обратимые операции (гейты), аргументы — константы inttoptr
  call void @__quantum__qis__h__body(ptr null)
  call void @__quantum__qis__cnot__body(ptr null, ptr inttoptr (i64 1 to ptr))
  br label %measurements

measurements:
  ; 3) необратимые операции (измерения), результат — writeonly ptr
  call void @__quantum__qis__mz__body(ptr null, ptr writeonly null)
  call void @__quantum__qis__mz__body(ptr inttoptr (i64 1 to ptr), ptr writeonly inttoptr (i64 1 to ptr))
  br label %output

output:
  ; 4) запись вывода и возврат кода выхода
  call void @__quantum__rt__tuple_record_output(i64 2, ptr @a_b)
  call void @__quantum__rt__result_record_output(ptr null, ptr @a)
  call void @__quantum__rt__result_record_output(ptr inttoptr (i64 1 to ptr), ptr @b)
  ret i64 0
}

attributes #0 = { "entry_point" "qir_profiles"="base_profile"
                  "output_labeling_schema"="schema_id"
                  "required_num_qubits"="2" "required_num_results"="2" }
attributes #1 = { "irreversible" }   ; на объявлении mz

!llvm.module.flags = !{!0, !1, !2, !3}
!0 = !{i32 1, !"qir_major_version", i32 2}
!1 = !{i32 7, !"qir_minor_version", i32 0}
!2 = !{i32 1, !"dynamic_qubit_management", i1 false}
!3 = !{i32 1, !"dynamic_result_management", i1 false}
```

Жёсткие требования Base Profile, которые задаёт этот шаблон:

- entry-point **не принимает аргументов и возвращает `i64`** (код выхода, `0` —
  успех);
- ровно четыре блока, соединённые **безусловными** `br`; обратимые операции — в
  `body`, необратимые (измерения) — в `measurements`, запись вывода — в `output`;
- единственные допустимые LLVM-инструкции: `call`, `br`, `ret`, `inttoptr`,
  `getelementptr inbounds` — никаких локальных вычислений;
- кубиты/результаты — константы `inttoptr (i64 N to ptr)`, индекс в
  `[0, required_num_*)`;
- функция измерения помечена `irreversible`, а её результат-параметр — `writeonly`.

## 7. Классическое управление и профили

У Adaptive Profile есть **обязательное ядро** (capabilities 1–4) и набор
**опциональных** возможностей (5–11), каждая из которых отражается отдельным
module flag (`Adaptive_Profile.md`). Это важно: «классическая арифметика» и
«ветвление по измерению» — это **разные** возможности, и арифметика — опциональна.

| Возможность | Base | Adaptive (обяз.) | Adaptive (опц.) |
|-------------|------|------------------|-----------------|
| Статическая нумерация кубитов/результатов | да | да | — |
| Гейты + измерение в конце, вывод через `record` | да | да | — |
| Измерение в середине схемы, использование кубита после | нет | да | — |
| `if read_result(r) { ... }` (forward-ветвление по `i1`) | нет | да | — |
| Классические вычисления (арифметика над int/float) | нет | нет | да (cap. 5) |
| IR-функции и их вызовы | нет | нет | да (cap. 6) |
| Циклы внутри `quant` (backward branching) | нет | нет | да (cap. 7) |
| Динамическое выделение кубитов/результатов | нет | нет | да (cap. 10) |
| Массивы кубитов/результатов | нет | нет | да (cap. 11) |

Реализация по этапам:

- **Этап 1 — Base Profile.** Линейная схема: инициализация → гейты → измерения →
  `record` (шаблон из §6a). Никакого ветвления и классики. Этого достаточно,
  чтобы заменить заглушку реальным лоуэрингом.
- **Этап 2 — Adaptive Profile (ядро).** Добавляется `read_result` → `i1` и `if`
  по результату (mid-circuit measurement, feed-forward); снимается требование
  «измерение только в конце».
- **Этап 3 — опциональные возможности Adaptive** по мере необходимости:
  классические вычисления (cap. 5) для вычисляемых углов/счётчиков, циклы
  (cap. 7), динамика и массивы (cap. 10–11).

Выбор профиля задаётся флагом CLI (расширение существующего `-q`/`--qir`),
например `--qir-profile=base|adaptive`, и проставляет соответствующие
`"qir_profiles"`, module flags профиля и флаги включённых опциональных
возможностей. Компилятор обязан **отклонять** программу, если она использует
возможность, не разрешённую выбранным профилем (§6.3).

## 9. Минимально жизнеспособный набор (MVP)

Чтобы получить **первую настоящую** трансляцию `quant` в QIR Base Profile,
достаточно:

1. Типы `qubit`, `result` + регистр `qubit[N]` (статический, `inttoptr`-нумерация).
2. Гейты: `h`, `x`, `z`, `cx` (этого хватает для Bell/GHZ).
3. Измерение `mz` (lowering в `void(ptr, writeonly ptr)`) и `record` (+
   `record_tuple` для нескольких результатов).
4. Лоуэринг в шаблон из §6a: четыре блока, `i64`-entry-point, автоматический
   подсчёт `required_num_qubits` / `required_num_results`, module flags v2 с
   `dynamic_*_management = false`.
5. Базовая проверка линейности (кубит измеряется ровно один раз и не
   используется после `mz`).

Сознательные ограничения MVP (всё это — **не** Base Profile, поэтому отклоняется
типчекером при `--qir-profile=base`): классические вычисления внутри `quant`,
вычисляемые углы, `if`/`read_result`, измерение в середине схемы, возврат
`result` из функции вместо `record`.

Всё остальное (повороты `rx/ry/rz`, `swap`, `ccx`, динамическое выделение,
Adaptive-ветвление и классика) — последующие итерации (§7, этапы 2–3).
